RevenueCatUI • Paywalls V2

Reactive paywalls that change with intent

Paywalls V2 already gives teams a flexible server-authored UI. The reactive state adapter makes that UI more precise: state changes update the fields that actually changed, apps can observe or gate those changes, and multiple paywalls can run side by side without stepping on each other.

Backward-compatible JSON Scoped state keys Propose → gate → commit Dynamic override fields Composable reducers
Adoption path Start with the existing Paywalls V2 payload and migrate behavior incrementally.
Developer promise Observe, reject, or replace UI state changes before they are committed.
User experience Update the package, text, or visibility that changed instead of rebuilding the world.

1. Executive summary

The adapter turns Paywalls V2 from a mostly static rendering pass into a scoped state pipeline. It keeps the backend JSON compatible, but gives the runtime a cleaner contract: propose a change, let the app decide, then commit exactly what changed.
  • Less redraw churn Field-level state lets SwiftUI do the work it is good at: updating small pieces of UI.
  • More app control Apps can gate package selection, personalization, or workflow-driven state without forking the paywall renderer.
  • Safer experimentation Runtime scope keeps copied paywalls and concurrent paywalls isolated, even when component IDs match.
The practical outcome: product teams can add more dynamic paywall behavior while SDK teams keep compatibility, debuggability, and ownership boundaries intact.

2. The problem this solves

State is too coarse

Today a package selection or condition change often causes full style recomputation. That is simple, but it makes the UI less predictable as paywalls become more dynamic.

Apps need a decision point

Some apps need to authenticate, enforce business rules, or adjust a selection before the paywall commits it. That should be explicit, not hidden in one-off callbacks.

Paywalls can be copied

Copied paywalls can keep identical component IDs. State propagation has to know which paywall instance it belongs to, not just which component ID changed.

The goal is not to make Paywalls V2 more complicated. It is to make state changes visible, reviewable, and scoped. That gives both the SDK and the consuming app clearer responsibilities.

3. What changes

From style snapshots

View models can still compute legacy styles, but migrated components also project the fields they render into state slots.

To committed field state

A text component can render committed text from the store while using the existing style path as initialization and fallback.

With proposed mutations

A change becomes a proposed event first. The app can accept it, reject it, or replace it. Only committed events update canonical state.

Core idea: the existing Paywalls V2 payload stays the source of truth, while the UI gets a finer-grained runtime projection of that payload.

4. System architecture

The adapter sits between decoded Paywalls V2 models and SwiftUI rendering. It compiles component identity, override property names, and runtime inputs into scoped state keys. Components can then render from committed state instead of recomputing the entire style tree.

5. Identity and state keys

State isolation is the foundation of the design. A key is not just a dashboard component ID. It is the combination of runtime paywall scope, observable component identity, and dynamic field name.

Runtime scope

PaywallStateScope ├─ instanceID: UUID ├─ paywallID: String? ├─ offeringIdentifier: String ├─ paywallRevision: Int? └─ workflowPageID: String?

Component identity

PaywallComponentIdentity ├─ paywallID: String? ├─ componentID: String ├─ type: String // metadata └─ name: String? // metadata

  A[PaywallStateScope] --> K[PaywallStateKey]
  B[PaywallComponentIdentity] --> K
  C[Field name] --> K
  K --> S[(PaywallStateStore)]

  A1["instanceID: runtime UUID"] --> A
  A2["paywallID: copied paywall boundary"] --> A
  A3["workflowPageID: workflow screen boundary"] --> A

  B1["componentID: backend id or deterministic legacy fallback"] --> B
  B2["type/name: useful metadata, not equality"] --> B

  C2["component.visible"] --> C
  C3["paywall.root_selected_package_id"] --> C
Risk How the architecture handles it
Copied paywalls keep identical component IDs. Observable identity includes paywall ID; mutable keys also include runtime instance scope.
The same paywall renders twice at once. instanceID separates the two live runtime trees.
Legacy payloads do not contain component IDs. The identity factory creates deterministic RevenueCat-reserved fallback IDs.

6. Mutation pipeline

The pipeline mirrors the iOS state example: requests become proposed changes, apps can gate those proposals, and only committed changes update canonical state and notify observers. 

sequenceDiagram
  participant UI as Component / Coordinator
  participant Store as PaywallStateStore
  participant Gate as App mutation hook
  participant State as Canonical state
  participant Events as Resolved events

  UI->>Store: request(PaywallStateMutation, details)
  Store->>Store: build PaywallStateChange.Event<Proposed>
  alt mutation hook exists
    Store->>Gate: proposal(change)
    Gate-->>Store: accept / reject / replace
  else no hook
    Store->>Store: accept by default
  end
  alt accepted or replaced
    Store->>State: atomically update value under lock
    Store->>Events: publish Event<Committed> on serialized queue
    Events-->>UI: update observed field
    Events-->>App: onPaywallStateChange
  else rejected
    Store-->>UI: no committed event
  end
Concurrency rule: canonical values live under lock. Combine subjects are for delivery, not synchronization.

7. Proposed vs committed state

The type system distinguishes changes that still need a decision from changes that are already resolved. Consumers cannot accidentally treat a proposal as a committed event. 

stateDiagram-v2
  [*] --> Requested
  Requested --> Proposed: create proposed event
  Proposed --> Rejected: reject()
  Proposed --> Accepted: accept()
  Proposed --> Replaced: replace
  Accepted --> Committed: commit original mutation
  Replaced --> Committed: commit replacement mutation
  Rejected --> [*]: no state change
  Committed --> [*]: publish committed event
PaywallStateChange.Event<PaywallStateChange.Proposed> PaywallStateChange.Event<PaywallStateChange.Committed> PaywallStateChange.Details // generic protocol; reducers cast to concrete details

8. Dynamic fields from existing JSON

The first adapter slice does not add new JSON fields. Instead, it preserves raw override properties and generates state fields from their serialized keys. This keeps the architecture composable while preserving existing payloads. 

flowchart LR
  O["override.properties"] --> Raw["rawProperties"]
  Raw --> Keys["PaywallOverridePropertyKeyBuilder"]
  Keys --> Direct["Direct component property field"]
  Raw --> Reducer["Reducer (future slices)"]
  Reducer --> Derived["Derived field"]
  Direct --> Mutations["State mutation list"]
  Derived --> Mutations
  Mutations --> Store["PaywallStateStore"]
Payload example State field Notes
{ "properties": { "visible": false } } component.visible Direct field from serialized property key.
{ "properties": { "textLid": "title" } } component.textLid Uses the component CodingKeys value; no manual casing conversion.
{ "properties": { "formats": ..., "iconName": ... } } Derived field in future icon slice Reducer can compose raw properties into a URL or display asset.
No re-encoding: raw properties are captured during the decode pass. The adapter does not recover them by encoding typed partials back into JSON.

9. Animation roadmap

Animation does not need to arrive all at once. The practical path is to start by animating the override model we already have, then graduate toward explicit paywall state once the runtime and dashboard are ready for a more expressive schema.

Phase 1: animate override groups

The bridge step keeps the current override model and adds animation metadata to the override itself. It is the quickest way to make selected package rows, badges, colors, and text feel smoother without asking the backend or dashboard to adopt a new state schema. 

{
  "conditions": [{ "type": "selected" }],
  "animation": {
    "type": "ease_in_out",
    "duration_ms": 300,
    "delay_ms": 0
  },
  "properties": {
    "textLid": "selected_title",
    "color": {
      "light": { "type": "hex", "value": "#2EBEE5" }
    },
    "visible": true
  }
}

Phase 2: make state explicit

The long-term step is closer to the POC architecture. The paywall owns named observable state values, while components declare anonymous rules that reference those state values in their when clauses. 

{
  "when": {
    "state": "selected_package_id",
    "operator": "=",
    "value": "annual"
  },
  "set": {
    "textLid": {
      "value": "annual_title",
      "animation": { "type": "instant" }
    },
    "color": {
      "value": { "light": { "type": "hex", "value": "#2EBEE5" } },
      "animation": { "type": "ease_in_out", "duration_ms": 250 }
    }
  }
}
flowchart LR
  State["Named paywall state"] --> Rule["Anonymous component rule"]
  Rule --> SetValues["Set field values"]
  SetValues --> FieldA["textLid animation"]
  SetValues --> FieldB["color animation"]
  FieldA --> Mutations["Independent mutations"]
  FieldB --> Mutations
  Mutations --> Store["Paywall state store"]
Step Why it comes first/next Tradeoff to carry forward
1. Override-level animation Fastest path. It reuses current conditions, current partial merging, and current Paywalls V2 JSON shape with a small additive animation field. Coarse-grained. One override may change text, color, visibility, spacing, and image values. One animation applies to all of those changes.
2. Explicit state rules Builds on the same reactive store, but moves from override patches to named observable paywall state and anonymous component rules. Requires a broader JSON evolution, validation rules, and cross-platform implementation work.

How grouping would work in Phase 2

Component rules do not need names of their own. They are anonymous projections from named paywall state to component values. If a rule has a group animation, values without their own animation can inherit it. Values with their own animation override the group. 

{
  "when": { "state": "selected_package_id", "operator": "=", "value": "annual" },
  "animation": { "type": "ease_in_out", "duration_ms": 300 },
  "set": {
    "visible": { "value": true },
    "color": { "value": { "light": { "type": "hex", "value": "#2EBEE5" } } },
    "textLid": {
      "value": "annual_title",
      "animation": { "type": "instant" }
    }
  }
}
Roadmap: animated overrides are the near-term bridge. Explicit state rules are the long-term model for truly dynamic paywalls and per-property animation.

10. Package selection flow

Package selection is the first high-value interaction routed through state. Package rows and tabs both request mutations. The selected package context updates only after the store publishes a committed event, so async gates and replace(with:) work correctly. 

sequenceDiagram
  participant Row as Package row / tabs
  participant Coord as PaywallPackageSelectionCoordinator
  participant Store as PaywallStateStore
  participant Gate as App mutation hook
  participant Context as PackageContext

  Row->>Coord: select package
  Coord->>Store: request root/sheet package mutation
  Store->>Gate: proposal
  Gate-->>Store: accept / reject / replace
  Store-->>Coord: committed selection event
  Coord->>Context: update(package, variableContext)

Sheet-specific behavior

stateDiagram-v2
  [*] --> RootSelection
  RootSelection --> SheetOpen: open sheet
  SheetOpen --> SheetSelection: select package in sheet
  SheetSelection --> SheetDismissed: dismiss sheet
  SheetOpen --> SheetDismissed: dismiss without selection
  SheetDismissed --> RootSelection: restore current workflow package or default

Sheet selection uses paywall.sheet[componentID].selected_package_id. Dismissal clears the sheet slot and restores the current workflow-selected package at dismissal time, not a stale package captured when the coordinator was created.

11. First component slice: Text

Text is the first component to project committed field state. It still computes a legacy style for initialization and fallback, but the rendered text can come from the committed component.textLid state slot. 

flowchart TD
  Env["Runtime inputs"] --> Style["Text component projection"]
  Overrides["Presented overrides and raw properties"] --> Style
  Style --> Projection["Projected text style"]
  Projection --> Mutations["Text field mutations"]
  Mutations --> Store["Paywall state store"]
  Store --> Publisher["Text key publisher"]
  Publisher --> View["Text component view"]
  Style --> Fallback["Legacy style fallback"]
  Fallback --> View

Why this is useful

If an app gate replaces or rejects a proposed text change, the view renders what the state store actually committed, not what the view model locally recomputed.

What comes next

The same pattern can expand to visibility, color, padding, package rows, tabs, icons, carousel selection, and other override-driven fields.
